# Implementation plan: Core game

## Overview

Browser-based implementation of Cadaco All-Star Baseball (later edition with K-O Dial + Strategy Disc). Solo play, data-driven player discs, SVG rendering with GSAP animation.

## Tech stack

- Vanilla JS with ES modules (native `import`/`export`, no bundler)
- GSAP for animation (CDN)
- SVG for all game graphics
- Vitest for game logic tests
- Playwright for UI/integration tests
- Python `http.server` for local dev
- JSON for player disc data

## Player disc JSON schema

```json
{
  "players": [
    {
      "id": "babe-ruth",
      "name": "Babe Ruth",
      "position": "outfield",
      "team": "Yankees",
      "era": "all-time",
      "sectors": [
        { "number": 1, "size": 28 },
        { "number": 7, "size": 15 },
        { "number": 10, "size": 42 },
        { "number": 14, "size": 30 }
      ]
    }
  ]
}
```

- `sectors`: array of { number, size } where number is 1-14 and size is the angular width in degrees (must sum to 360)
- `position`: one of "pitcher", "catcher", "first-base", "second-base", "shortstop", "third-base", "outfield"
- Disc color derived from position (white center = pitcher/catcher, grey = infielder, red = outfielder)

## Phase 2: Game rules engine

All rules logic is pure functions with no DOM dependencies. Fully testable in isolation.

### 2.1 Rules constants — `js/game/rules.js`

Define the three lookup tables as exported constants:

**BATTING_KEY**: Map of number (1-14) to result type
```js
export const BATTING_KEY = {
  1: 'home-run',
  2: 'ground-ball',
  3: 'fly-ball',
  4: 'fly-ball',
  5: 'triple',
  6: 'ground-ball',
  7: 'single',
  8: 'fly-ball',
  9: 'walk',
  10: 'strikeout',
  11: 'double',
  12: 'ground-ball',
  13: 'single',
  14: 'fly-ball'
}
```

**KO_DIAL**: Map of (letter, resultType) to outcome object describing batter and runner effects.

**STRATEGY**: Map of (play type, letter) to outcome object.

Also define which result types require a K-O Dial spin: `single`, `fly-ball`, `ground-ball`.

**Test:** Verify all 14 batting key entries resolve. Verify K-O table has 5 entries (K-O) for each of the 3 types. Verify strategy tables have 5 entries each for all 7 play types.

### 2.2 Batting resolution — `js/game/batting.js`

```js
export function resolveBatting(sectorNumber)
```

- Takes a sector number (1-14)
- Returns `{ type, needsKoDial }` where type is the result string and needsKoDial is boolean
- Home runs, triples, doubles, walks, strikeouts resolve immediately
- Singles, fly balls, ground balls need K-O Dial follow-up

**Test (TDD):**
- Sector 1 → home-run, no K-O needed
- Sector 7 → single, K-O needed
- Sector 10 → strikeout, no K-O needed
- All 14 sectors produce valid results

### 2.3 K-O Dial resolution — `js/game/ko-dial.js`

```js
export function resolveKoDial(letter, resultType, baseState)
```

- Takes the K-O letter (K/L/M/N/O), the result type (single/fly-ball/ground-ball), and current base state
- Returns `{ batter: { base, out }, runners: [{ from, to, out }], description }`
- Base state is `{ first: bool, second: bool, third: bool }`

**Test (TDD):**
- Fly ball + K → batter safe at 2B on error, runners advance 2
- Fly ball + O → fly out, lead runner doubled off
- Single + L → safe at 1B, runners advance 1
- Ground ball + M → out at 1B, lead runner out at 2B (DP)
- Ground ball + O with bases loaded → out at 1B, runner out at plate
- All 15 combinations (5 letters x 3 types) produce valid outcomes

### 2.4 Baserunning — `js/game/baserunning.js`

```js
export function advanceRunners(baseState, outcome)
```

- Takes current base state and the resolved outcome (from batting or K-O)
- Returns `{ newBaseState, runsScored, outsRecorded, details[] }`
- Handles: forced advances, tag-up advances, double plays, runner thrown out at plate, errors
- Special cases: home run (clear bases + batter scores), triple (clear bases + batter to 3rd), double (batter to 2nd + advance per K-O), walk (advance only if forced)

**Test (TDD):**
- Home run with bases loaded → 4 runs, bases empty
- Walk with runner on 1B only → runners on 1B and 2B, 0 runs
- Walk with bases loaded → 1 run, bases still loaded
- Double play: runner on 1B, ground ball M → 2 outs, bases empty
- Fly ball sacrifice: runner on 3B, fly ball L → 1 run, 1 out
- Single with runners on 1B and 3B, K-O N → runner scores from 3B, runner to 3B from 1B, batter at 1B

### 2.5 Strategy plays — `js/game/strategy.js`

```js
export function getAvailableStrategies(baseState)
export function resolveStrategy(playType, letter, baseState)
```

- `getAvailableStrategies`: given base state, returns which strategy plays are legal (e.g., squeeze requires runner on 3B, steal from 1B requires runner on 1B)
- `resolveStrategy`: given the play type, strategy letter, and base state, returns outcome in the same format as K-O resolution

**Strategy play types and their base requirements:**
- Single steal from 1B: runner on 1B (A-E)
- Double steal 1B+3B: runners on 1B and 3B (A-E)
- Hit and run: runner on 1B (A-E)
- Squeeze: runner on 3B (A-E)
- Single steal from 2B: runner on 2B (F-J)
- Double steal 1B+2B: runners on 1B and 2B (F-J)
- Sacrifice bunt from 1B: runner on 1B (F-J)
- Sacrifice bunt from 2B: runner on 2B (F-J)

**Test (TDD):**
- No runners → no strategies available
- Runner on 1B only → steal-1b, hit-and-run, sac-bunt-1b available
- Runner on 3B only → squeeze available
- Squeeze + A → grounds out, runners advance 1 (runner scores)
- Squeeze + E → pops out, runner caught off 3B (DP)
- All strategy chart entries produce valid outcomes

### 2.6 Game state machine — `js/game/state.js`

```js
export function createGame(homeLineup, visitorLineup)
export function processAtBat(game, result)
export function processKoDial(game, letter)
export function processStrategy(game, playType, letter)
export function intentionalWalk(game)
```

Game state object:
```js
{
  inning: 1,
  halfInning: 'top',       // 'top' or 'bottom'
  outs: 0,
  bases: { first: null, second: null, third: null },
  score: { home: [0,0,...], visitor: [0,0,...] },
  homeLineup: [],           // array of player disc objects
  visitorLineup: [],
  homeBatterIndex: 0,
  visitorBatterIndex: 0,
  phase: 'batting',         // 'batting' | 'ko-dial' | 'strategy' | 'game-over'
  pendingResult: null,       // stored between batting spin and K-O spin
  log: []                   // play-by-play entries
}
```

State transitions:
- `batting` → spin resolves to immediate result (HR/triple/double/walk/K) → record result, advance batter index, check 3 outs → stay in `batting` or flip half-inning
- `batting` → spin resolves to single/fly/ground → store pending, move to `ko-dial`
- `ko-dial` → K-O spin resolves full outcome → record, back to `batting`
- `batting` → player calls strategy → move to `strategy`
- `strategy` → strategy spin resolves → record, back to `batting`
- End of 9th (or extra innings) with lead → `game-over`

**Test (TDD):**
- New game starts at top of 1st, 0 outs, bases empty
- 3 strikeouts → flip to bottom of 1st
- Home run in bottom of 9th with home team trailing by 1 → walk-off, game over
- Batter index wraps around after 9th batter
- Intentional walk with bases loaded scores a run
- Full 9-inning game simulation with predetermined spins produces correct final score

## Phase 3: Player disc data

### 3.1 Disc loader and validator — `js/data/players.js`

```js
export async function loadPlayers(url)
export function validateDisc(disc)
```

- `loadPlayers`: fetch JSON, validate each disc, return array
- `validateDisc`: verify sectors sum to 360, all sector numbers are 1-14, required fields present, position is valid

**Test (TDD):**
- Valid disc passes validation
- Sectors not summing to 360 → error
- Missing name → error
- Invalid position → error
- Duplicate sector numbers → error

### 3.2 Starter player set — `data/players.json`

Create ~20 players covering all positions, enough for two 9-player teams with a few bench options. Use a mix of all-time greats with varied statistical profiles:

- Power hitters (large HR sectors): e.g., Babe Ruth, Hank Aaron
- Contact hitters (large single sectors, small K): e.g., Rod Carew, Tony Gwynn
- Balanced: e.g., Willie Mays, Roberto Clemente
- Pitchers (large K/ground ball sectors): e.g., Nolan Ryan, Sandy Koufax, Tom Seaver, Bob Gibson
- Catchers: e.g., Johnny Bench, Yogi Berra

Sector sizes will be approximated from known batting statistics to feel right, not claiming exact historical accuracy.

**Verify:** All discs pass validation. Load in browser and log player count.

### 3.3 Spinner probability — `js/game/batting.js`

```js
export function spin(disc)
```

- Given a player disc, randomly select a sector weighted by angular size
- Return the sector number
- Uses `Math.random()` scaled to 360 degrees, then walks sectors to find which one the "spinner" landed in

**Test (TDD):**
- Disc with single sector of 360° always returns that number
- Disc with two equal 180° sectors returns each roughly 50% over many trials (statistical test)
- Result is always a valid sector number from the disc

### 4.3 Spinner component — `js/ui/spinner.js`

```js

export function spinSpinner(spinner, targetAngle, onComplete)
```

- SVG spinner assembly: outer K-O letter ring (red), inner blue rectangle, metal arrow
- `spinSpinner` uses GSAP to animate:
  - Multiple full rotations + final position
  - Physics-based easing: fast start, gradual deceleration, slight overshoot, settle
  - Duration ~2-3 seconds
  - `onComplete` callback fires with the result (sector number for batting, K-O letter for fielding)
- K-O letter ring: 5 letters (K, L, M, N, O) in sectors around the outer ring, read by arrow position after spin
- Player disc sits visually "under" the spinner arrow

**Verify:** Click to spin, arrow rotates with satisfying physics, lands on a sector, result logged.

## Phase 5: Game UI and flow

### 5.1 Draft / team selection — `js/ui/lineup.js`

```js

```

- Display available player discs grouped by position
- Alternating pick order per the box lid rules (position-based draft chart)

- Show both teams' rosters building up
- After draft, arrange batting order (drag to reorder or click-to-swap)
- "Start Game" button when both lineups are set

**Verify (Playwright):** Complete a full draft, set batting orders, start game.

### 5.2 Game controls — `js/ui/controls.js`

```js
export function createControls(container, callbacks)
```

- **Spin button**: prominent, triggers batting spin (disabled during animation or when waiting for K-O)
- **Strategy menu**: dropdown/popover showing available strategy plays (context-sensitive based on base state), disabled when no runners
- **Intentional walk button**: available to fielding side
- **Substitution**: swap a player disc from bench into lineup
- **Current phase indicator**: "Batting" / "K-O Dial" / "Strategy"
- Controls disable/enable based on game phase

**Verify (Playwright):** Spin button triggers animation, strategy menu shows correct options for base state.

### 5.3 Main game loop integration — `js/main.js`

Wire everything together:

1. Load player data
2. Run draft UI
3. Create game state with selected lineups
4. Render board, scoreboard, controls
5. Game loop:
   - Player clicks Spin → `spin(currentBatter)` → `resolveBatting()` → animate spinner to result
   - If K-O needed → prompt for K-O spin → `resolveKoDial()` → animate fielding spinner
   - Apply outcome via `advanceRunners()` → `processAtBat()` → update state
   - Animate baserunners, update scoreboard, narrate
   - Check for inning change, game over
   - Advance to next batter
6. On game over → show final score, option to play again

**Verify (Playwright):** Play through a complete game from draft to final score.

## Phase order and dependencies

```
Phase 1 (scaffolding) ── no dependencies
Phase 2 (rules engine) ── no dependencies, can parallel with Phase 1
Phase 3 (player data)  ── depends on Phase 2 (batting.js spin function)
Phase 4 (visual layer)  ── depends on Phase 1 (HTML shell)
Phase 5 (game UI)       ── depends on Phases 2, 3, 4
Phase 6 (strategy UI)   ── depends on Phases 2, 5
```

Phases 1 and 2 can be built in parallel. Phase 3 can start as soon as Phase 2's data format is defined. Phase 4 can start as soon as Phase 1 is done. Phases 5 and 6 are integrations that bring everything together.
